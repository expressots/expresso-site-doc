# Containerize

The `containerize` command (alias: `c`) inspects your project and
emits production-ready container artifacts: a multi-stage
`Dockerfile`, a hot-reload `Dockerfile.development`, a
`docker-compose.yml`, a `.dockerignore`, and (optionally) a Kubernetes
manifest set or a CD GitHub Actions workflow.

It is **idempotent** — re-running it overwrites the generated files
in place, so you can keep tweaking flags until the output matches
what you need, then customize the files freely.

## Quick start

```bash
expressots containerize
# equivalent to:
#   expressots containerize docker production --preset standard --analyze
```

This single command will:

1. Analyze your `package.json`, `tsconfig.build.json`, lock file, and
   bootstrap config.
2. Write `Dockerfile` (multi-stage prod) and `Dockerfile.development`
   (with hot reload).
3. Write `docker-compose.yml` (and a development variant when you
   target development).
4. Write `.dockerignore`.

```bash
docker build -t myapp .
docker run -p 3000:3000 myapp
# or
docker compose up
```

## Synopsis

```bash
expressots containerize [target] [environment] [flags]
```

### Targets

| Target       | What gets generated                                                  |
| ------------ | -------------------------------------------------------------------- |
| `docker`     | Dockerfile(s) + (optionally) `docker-compose.yml`. **Default.**      |
| `kubernetes` | Dockerfile(s) **plus** `k8s/deployment.yaml`, `service.yaml`, `configmap.yaml`. Alias: `k8s`. |
| `compose`    | Dockerfile(s) **plus** `docker-compose.yml`. The Dockerfile is required because the compose file uses `build:` context. |

### Environments

| Environment  | Effect                                                                |
| ------------ | --------------------------------------------------------------------- |
| `production` | Multi-stage build, devDeps pruned, `NODE_ENV=production`. **Default.** |
| `staging`    | Same multi-stage shape as production, but with `NODE_ENV=staging`. Output file is `Dockerfile.staging`. |
| `development`| Single-stage Dockerfile with hot reload, debug port `9229`, mounts `./src` into the container. **Also generates a production Dockerfile** so you have both. |
| `all`        | Same as passing `development` — emits both dev and prod Dockerfiles.  |

## Flags

| Flag                       | Default          | Description                                                                                       |
| -------------------------- | ---------------- | ------------------------------------------------------------------------------------------------- |
| `--preset`                 | `standard`       | One of `minimal`, `standard`, `secure`, `fast-startup`, `dev`, `multi-arch`. Controls base image, security hardening, and health checks. |
| `--analyze`                | `true`           | Inspect the project before generation. Disable with `--no-analyze` to use defaults.               |
| `--skip-compose`           | `false`          | Skip writing `docker-compose.yml` when target is `docker`.                                        |
| `--include-ci`             | `false`          | Also write a GitHub Actions **CD** workflow (`.github/workflows/cd-docker.yml`). See [CD vs CI](#cd-vs-ci-pipelines) below. |
| `--ci-platform`            | `github`         | Platform for `--include-ci`. One of `github`, `gitlab`, `circleci`, `jenkins`, `bitbucket`, `azure`, `all`. |
| `--ci-strategy`            | `comprehensive`  | One of `basic`, `comprehensive`, `security-focused`. Controls which jobs are included.            |
| `--include-security-scans` | `true`           | Add Trivy / Snyk image scanning jobs.                                                             |
| `--include-e2e`            | `false`          | Add an E2E test job that runs against the built image.                                            |

## What gets detected

`expressots containerize --analyze` reads the following from your
project to drive code generation:

- **Node version** — from `engines.node` in `package.json`, falling
  back to the runtime's version.
- **Package manager** — from the lock file. `npm`, `pnpm`, `yarn`,
  and `bun` are all supported. The generated Dockerfile and CI
  install/build/dev commands will match.
- **Entry point** — from `tsconfig.build.json` (preferred) or
  `tsconfig.json`. The `CMD` of the production Dockerfile will point
  at the right compiled file under your `outDir`.
- **Database / Redis dependencies** — adds matching services to
  `docker-compose.yml`.
- **Health check paths** — controllers exposing `/health`, `/ready`,
  or `/live` are wired into the Dockerfile `HEALTHCHECK` and the
  Kubernetes liveness/readiness probes.
- **Local `file:` dependencies** — generates a `docker-setup.js`
  helper and a `package.docker.json` so Docker builds can resolve
  monorepo paths. Add `npm run docker:setup && docker build` (or the
  pnpm/yarn/bun equivalent generated for you) to your release flow.
- **Bootstrap env config** — if your `main.ts` uses
  `envFileConfig({...})`, the `.dockerignore` and `Dockerfile` will
  copy the right `.env.<environment>` into the image; warnings are
  printed when required env files are missing.

## Examples

```bash
# Bare-minimum production Docker image (ignore project analysis):
expressots containerize docker production --no-analyze --preset minimal

# Hardened image with non-root user + healthcheck:
expressots containerize docker production --preset secure

# Kubernetes manifests for a 3-replica deployment:
expressots containerize k8s production

# Staging variant (NODE_ENV=staging, multi-stage build):
expressots containerize docker staging

# Both dev and prod Dockerfiles plus a compose file with hot reload:
expressots containerize docker development

# Add a GitHub Actions CD workflow alongside the Dockerfile:
expressots containerize docker production --include-ci --ci-platform github
```

## CD vs CI pipelines

ExpressoTS ships **two complementary** pipeline generators:

| Generator                              | Output (GitHub example)              | Scope                                   |
| -------------------------------------- | ------------------------------------ | --------------------------------------- |
| `expressots cicd init`                 | `.github/workflows/ci.yml`           | **CI** — lint, unit tests, coverage     |
| `expressots containerize --include-ci` | `.github/workflows/cd-docker.yml`    | **CD** — Docker build, push, deploy     |

On **GitHub Actions** both files coexist happily. On platforms that
only allow a single canonical pipeline file (GitLab CI, Jenkins,
CircleCI, Bitbucket, Azure DevOps), the CLI prints a warning when
`containerize --include-ci` would overwrite `cicd init`'s output —
prefer `cicd init` for those platforms.

## Tips

- The `--analyze` step prints recommendations about your bootstrap
  config (e.g. "switch to `skipFileLoading: true` for a cleaner
  container build"). Read them.
- The generated files are **yours** to edit. Re-running `containerize`
  will overwrite them, so keep customizations in version control or
  copy them out before re-running.
- For monorepo / `file:` dependencies, run
  `npm run docker:setup` before `docker build`. The CLI generates
  this script for you and writes a `docker:build` composite to
  `package.json` that does both for the package manager you use.
