# `@expressots/studio`

Studio is the browser dashboard you launch with `expressots studio`. It connects (over a WebSocket) to a running ExpressoTS process and shows you, in real time:

- **Routes**: every controller method, with HTTP verb, path, version, regex constraint, guards, interceptors, and content-negotiation metadata.
- **Providers**: every `@provide`d class, its scope, lifecycle hooks, health, and metrics.
- **Middleware**: the resolved pipeline order, including built-ins from the active preset.
- **Lifecycle**: `IBootstrap` / `IShutdown` hooks, with status and timing.
- **Events**: registered handlers, fire counts, and the last payload (when introspectable).
- **Logs**: scoped log stream from the framework `Logger`.

It's an explicit dev-time tool. **Don't ship it to production**.

## Installing

You don't install Studio manually. The [`@expressots/cli`](./cli.mdx) does it for you when you run `expressots studio`:

```bash
expressots studio
# 🐎 ExpressoTS CLI
# 🟢 Studio is running at http://localhost:8585
```

## How it talks to your app

Studio is a passive front-end. The runtime data comes from [`@expressots/studio-agent`](./studio-agent.mdx), which is mounted in your application by `AppExpress` whenever:

- `process.env.NODE_ENV !== "production"`, **or**
- you call `setupExpressoTSStudioAgent()` explicitly.

The agent exposes a WebSocket endpoint that streams runtime events. Studio connects to it and renders them.

## Production safety

Studio is **never enabled in production** by the framework. The agent itself is the gatekeeper. It inspects `NODE_ENV` and refuses to bind unless explicitly forced. See [Studio agent → Security](./studio-agent.mdx#security).

If you want to inspect a remote staging app, run Studio locally and tunnel the agent's WebSocket port (e.g. `ssh -L 8585:localhost:8585 staging-host`). Don't expose the agent on a public port.
