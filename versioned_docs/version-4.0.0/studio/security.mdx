# Security

Studio's Security view is a local, always-on security analyst for your application. It combines two complementary lenses:

1. **Supply-chain analysis**: every dependency in your lockfile, cross-referenced against the npm registry advisory database and OSV.dev, with transitive root-cause chains, reachability scoring, and one-click fixes.
2. **Runtime posture analysis**: your recorded HTTP traffic, routes, logs, and DI graph scored against the OWASP API Security Top 10.

The result is a single score letter (A–F) on the Status dashboard, and a drill-down view with copy-pasteable remediation commands.

:::note Local only
Studio never sends your `package.json`, your `package-lock.json`, your code, or your traffic anywhere. `npm audit` and OSV lookups happen from your machine.
:::

## Supply-chain analysis

### How findings are gathered

When you open the Security view (or every time the lockfile changes), the agent:

1. Spawns `npm audit --json` in your project root as an async child process.
2. Cross-references the result against [OSV.dev](https://osv.dev/) for non-npm-registry advisories and richer metadata.
3. Parses `package-lock.json` v2/v3 into an in-memory dependency graph (BFS over `dependencies`/`requires`).
4. Builds a runtime reachability snapshot from your `src/` imports, DI graph, and recorded exchanges.
5. Caches the merged report on disk so a re-scan only re-runs `npm audit` when the lockfile mtime changes.

The full scan stays off the request hot path; the UI is broadcast a single debounced `security` frame when results are ready.

### Anatomy of a finding

Each advisory in the UI shows:

| Field              | Meaning                                                                                                        |
| ------------------ | -------------------------------------------------------------------------------------------------------------- |
| Severity           | `critical` / `high` / `moderate` / `low`, sorted top-down.                                                     |
| Package + version  | The exact installed version that's affected.                                                                   |
| Advisory IDs       | GHSA, CVE, and OSV IDs with clickable links to the upstream report.                                            |
| Root cause         | The shortest dependency chain from one of your direct dependencies to the vulnerable package.                  |
| Reachability       | `confirmed` / `likely` / `unreachable` / `unknown` (see below).                                                |
| Fix command        | A copy-pasteable `npm` command, plus an **Apply fix** button that runs it for you.                             |

### Reachability scoring

Reachability is what turns a 200-item advisory list into the 8 things you actually need to fix this week.

For every vulnerable package, the agent asks "is this code path actually exercised at runtime?":

| Reachability    | What it means                                                                                                                       |
| --------------- | ----------------------------------------------------------------------------------------------------------------------------------- |
| `confirmed`     | At least one recorded exchange hit a route whose source code (transitively) imports the vulnerable package.                         |
| `likely`        | A route's source imports the package, but no recorded exchange has hit that route yet.                                              |
| `unreachable`   | The package is in the lockfile but no source file in `src/` imports it (e.g. a transitive dev-only dependency).                     |
| `unknown`       | Recording disabled or the source-scan timed out.                                                                                    |

`unreachable` findings are de-prioritized in the score and never block the **Recommended fixes** rollup.

### Root-cause chains

Most `npm audit` findings are transitive. The agent walks the lockfile graph to show **which direct dependency dragged it in**:

```text
expressots-app
└── @nestjs/swagger@8.1.0           ← direct dep
    └── swagger-ui-express@5.0.1     ← transitive
        └── send@0.18.0              ← vulnerable
```

When a fix requires upgrading `@nestjs/swagger`, the Security view says so explicitly, instead of leaving you to figure out why `npm audit fix` keeps saying "force" is required.

### Recommended fixes

Findings that share an upgrade target are grouped into a single **FixGroup** at the top of the page. Each group shows:

- The package + version to upgrade.
- The list of advisories it closes (e.g. "Fixes CVE-2024-1234 and 2 more").
- Whether the upgrade is a SemVer-major bump (warning shown).
- A single **Apply fix** button that runs the command for the whole group.

Under the hood, each finding resolves to one of these fix kinds:

| Fix kind            | Command shape                                  | When it's chosen                                                       |
| ------------------- | ---------------------------------------------- | ---------------------------------------------------------------------- |
| `install`           | `npm install pkg@x.y.z`                        | npm reports a specific patched version.                                |
| `audit-fix`         | `npm audit fix`                                | npm says it can auto-fix without breaking changes.                     |
| `audit-fix-force`   | `npm audit fix --force`                        | npm says it can fix but the upgrade is a SemVer-major bump.            |
| `override`          | `npm install` after editing `overrides`        | The vulnerable package is a transitive dep with no patched parent yet. |
| `none`              | (no command)                                   | No published fix exists yet (you'll see a "Watch for updates" note).   |

### Applying a fix

Clicking **Apply fix** in the UI:

1. Streams the live `npm` output line-by-line into a terminal-style banner at the top of the view.
2. Locks all other fix buttons until the run completes (one fix at a time, intentionally).
3. Re-runs `npm audit` + OSV automatically when the run finishes, so the report updates without a manual refresh.
4. Reports success/failure plus the exit code in the UI.

The fix runner uses an **allow-listed argv**: only `npm install <pkg>@<version>` and `npm audit fix [--force]` are accepted. There is no shell interpolation.

## Runtime posture analysis

Recording HTTP traffic gives Studio something `npm audit` can't see: how your app behaves at runtime. The posture analyzer scores recorded exchanges, routes, logs, and the DI graph against the **OWASP API Security Top 10** categories:

| OWASP category                     | What Studio looks for                                                                                  |
| ---------------------------------- | ------------------------------------------------------------------------------------------------------ |
| API1: BOLA                         | Routes with `:id`-style params that respond without an `AuthenticatedGuard` or owner check in the DI graph |
| API2: Broken Authentication        | Sensitive routes hit with no `Authorization` header in the recording                                   |
| API3: Excessive Data Exposure      | Response bodies that leak fields with sensitive-looking names (`password`, `apiKey`, `token`)          |
| API4: Unrestricted Resource Use    | Routes that respond > 2 s or that have no `TimeoutInterceptor` in their pipeline                       |
| API5: Broken Function Authz        | Routes with `admin`-shaped paths that don't require an admin role                                      |
| API7: Server-Side Misconfig        | `app.use()` middleware mistakes (e.g. CORS `*`, missing helmet)                                        |
| API8: Injection                    | Error logs that contain raw SQL / shell / NoSQL exception patterns                                     |
| API9: Improper Inventory           | Routes detected by source-scan that never appear in recordings (zombie routes)                         |
| API10: Unsafe Consumption of APIs  | Outgoing HTTP calls captured in logs without TLS verification                                          |

Each finding has a severity, an explanation, and a deep link into the recorded exchange that triggered it.

:::caution Heuristics, not guarantees
Posture findings are **heuristic**. They are very useful as a development-time second opinion, but they are not a replacement for a security review. A `confirmed` supply-chain CVE always outranks a heuristic posture flag in the aggregate score.
:::

## Score formula

The letter grade on the Status dashboard is a weighted aggregate:

```text
score = 100
        − 25 × critical_reachable
        − 12 × high_reachable
        −  5 × moderate_reachable
        −  1 × low_reachable
        −  8 × critical_runtime
        −  3 × high_runtime
        −  1 × moderate_runtime

grade = A (≥90)  B (≥75)  C (≥60)  D (≥40)  F (<40)
```

`unreachable` findings count for ⅕ of their reachable equivalent. A clean codebase with only dev-only `unreachable` advisories will still score an A.

## Performance

| Concern                | Mitigation                                                                                              |
| ---------------------- | ------------------------------------------------------------------------------------------------------- |
| `npm audit` cost       | Runs in a child process, never on the request thread.                                                   |
| Re-scan frequency      | Triggered only by lockfile mtime changes, fix completions, or a manual rescan from the UI.              |
| OSV lookups            | Cached on disk in `.studio/osv-cache.json` with a TTL.                                                  |
| UI traffic             | A single debounced `security` frame; suppressed entirely when no client is connected.                   |
| Production             | The studio-agent module is never imported when `NODE_ENV=production`. Zero production overhead.        |
