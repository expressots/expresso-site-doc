# Bootstrap

The `bootstrap()` function provides zero-configuration application startup with comprehensive options for different deployment scenarios.

## Basic Usage

The simplest way to start your application:

```typescript
import { bootstrap } from "@expressots/core";
import { App } from "./app";

bootstrap(App);
```

This starts the application on port 3000 (or `process.env.PORT` if set) with automatic environment detection.

## Configuration Options

### Port Configuration

```typescript
// Custom port
bootstrap(App, { port: 8080 });

// OS-assigned port (useful for testing)
const webServer = await bootstrap(App, { port: 0 });
const actualPort = await webServer.getPort(); // Use in tests
```

Port priority: `options.port` > `process.env.PORT` > `3000`

### Application Metadata

```typescript
await bootstrap(App, {
    appName: "My API",
    appVersion: "1.0.0",
});
```

Defaults to `package.json` name and version if not specified.

### Environment Configuration

The `envFileConfig` option enables .env file loading (opt-in):

```typescript
await bootstrap(App, {
    currentEnvironment: "development",
    envFileConfig: {
        files: {
            development: ".env.dev",
            production: ".env.prod",
            staging: ".env.staging",
        },
        required: ["DATABASE_URL", "JWT_SECRET"],
        autoCreateTemplate: true,
        validateValues: true,
    },
});
```

**Environment File Options:**

-   `files`: Custom mapping of environment names to .env file paths
-   `required`: Array of required environment variable names
-   `autoCreateTemplate`: Automatically create template .env file if missing
-   `validateValues`: Validate that all variables have non-empty values
-   `skipFileLoading`: Skip .env file loading (use `process.env` only)
-   `ciMode`: Force CI/CD mode (skips file loading, validates process.env)

### Type-Safe Configuration

You can scaffold type-safe configuration files using the CLI:

```bash
expressots generate config <name>
# or shorthand
expressots g cfg <name>
```

This generates a `config.ts` file with type-safe configuration using `defineConfig`:

```typescript
import { defineConfig, Env, loadEnvSync } from "@expressots/core";

// Load environment files before config resolution
const envFiles = {
    development: ".env.local",
    production: ".env.prod",
};

loadEnvSync({ files: envFiles });

export const appConfig = defineConfig({
    app: {
        name: Env.string("APP_NAME", { default: "My App" }),
        version: Env.string("APP_VERSION", { default: "1.0.0" }),
    },
    server: {
        port: Env.number("PORT", { default: 3000 }),
    },
    database: {
        url: Env.string("DATABASE_URL", { required: true }),
        pool: Env.number("DB_POOL_SIZE", { default: 10, min: 1, max: 100 }),
    },
    bootstrap: {
        envFileConfig: {
            autoCreateTemplate: true,
            files: envFiles,
        },
    },
});

// Export typed config values
export const config = appConfig.values;
export type AppConfig = typeof config;
```

**Benefits of Type-Safe Configuration:**

-   Full TypeScript inference for all config values
-   Validation with helpful error messages
-   Default values and constraints (min, max, etc.)
-   Secret management with automatic redaction
-   Multi-environment support
-   Can be integrated with bootstrap options

You can then use the configuration in your bootstrap call:

```typescript
import { bootstrap } from "@expressots/core";
import { App } from "./app";
import { appConfig } from "./config";

await bootstrap(App, {
    port: appConfig.values.server.port,
    appName: appConfig.values.app.name,
    appVersion: appConfig.values.app.version,
    envFileConfig: appConfig.values.bootstrap.envFileConfig,
});
```

## Common Patterns

### Development with Auto-Creation

```typescript
await bootstrap(App, {
    currentEnvironment: "development",
    envFileConfig: {
        autoCreateTemplate: true,
        required: ["DATABASE_URL"],
    },
});
```

### Production with Validation

```typescript
await bootstrap(App, {
    currentEnvironment: "production",
    envFileConfig: {
        files: {
            production: ".env.prod",
        },
        required: ["DATABASE_URL", "JWT_SECRET", "API_KEY"],
        validateValues: true,
    },
});
```

### Containerized Deployment (Docker/K8s)

```typescript
await bootstrap(App, {
    envFileConfig: {
        skipFileLoading: true, // Use process.env only
        required: ["DATABASE_URL", "REDIS_URL"],
    },
});
```

### Testing with Dynamic Ports

```typescript
const webServer = await bootstrap(App, {
    port: 0, // Auto-assign available port
    envFileConfig: {
        skipFileLoading: true,
    },
});

// Get the actual assigned port
const actualPort = await webServer.getPort();
expect(actualPort).toBeGreaterThan(0);

// Or access the HTTP server directly for advanced use cases
const httpServer = await webServer.getHttpServer();
```

## Environment Behavior

ExpressoTS recognizes the following environments: `development`, `staging`, `production`, `test`, or any custom string. The current environment is determined by:

1. `currentEnvironment` option in `bootstrap()`
2. `process.env.NODE_ENV`
3. Defaults to `"development"`

### Automatic Framework Behavior

The framework automatically adjusts behavior based on the environment:

#### Error Handling

**Development & Test:**

-   Full error messages are exposed to clients
-   Stack traces are included in error responses
-   Detailed error context for debugging

**Production & Staging:**

-   Generic error messages ("An unexpected error occurred")
-   Stack traces are hidden from responses (but still logged server-side)
-   Internal implementation details are protected

```typescript
// Example error response in development
{
  "title": "User not found with id: 123",
  "status": 404,
  "stack": "Error: User not found...\n    at UserService.findById..."
}

// Example error response in production
{
  "title": "An unexpected error occurred",
  "status": 500
  // No stack trace
}
```

#### Environment Validation

**Production:**

-   `validateValues` defaults to `true`
-   Missing required environment variables will fail startup
-   Strict validation is enforced

**Other environments:**

-   `validateValues` defaults to `false`
-   Missing variables produce warnings instead of errors
-   Allows for more flexible local development

#### Helper Method

Use `isDevelopment()` in your application to conditionally execute code:

```typescript
// In your App class (extends AppExpress)
async configureServices(): Promise<void> {
    this.Middleware.setErrorHandler({
        showStackTrace: await this.isDevelopment(),
    });
}
```

### Environment-Specific Configuration

Use `defineConfig` to set different defaults per environment:

```typescript
import { defineConfig, Env } from "@expressots/core";

export const config = defineConfig({
    server: {
        host: Env.string("HOST", {
            development: "localhost",
            staging: "0.0.0.0",
            production: "0.0.0.0",
        }),
    },

    app: {
        logLevel: Env.enum("LOG_LEVEL", ["debug", "info", "warn", "error"], {
            development: "debug",
            staging: "info",
            production: "warn",
        }),

        debug: Env.boolean("DEBUG", {
            development: true,
            staging: false,
            production: false,
        }),
    },

    database: {
        url: Env.url("DATABASE_URL", {
            development: "postgresql://localhost:5432/dev_db",
            staging: undefined, // Must be set explicitly
            production: undefined, // Must be set explicitly
        }),

        ssl: Env.boolean("DB_SSL", {
            development: false,
            staging: true,
            production: true,
        }),
    },
});
```

The framework automatically selects the correct default based on `NODE_ENV`.

### Environment File Loading

Each environment loads its corresponding `.env` file by convention:

| Environment   | Default File       | `validateValues` Default |
| ------------- | ------------------ | ------------------------ |
| `development` | `.env.development` | `false`                  |
| `staging`     | `.env.staging`     | `false`                  |
| `production`  | `.env.production`  | `true`                   |
| `test`        | `.env.test`        | `false`                  |
| CI/CD         | `process.env` only | `true`                   |

Optional files are loaded silently (in order): `.env`, `.env.local`, `.env.{environment}.local`

:::tip Testing Best Practice
For tests, use `skipFileLoading: true` to avoid file I/O and rely on `process.env`:

```typescript
await bootstrap(App, {
    envFileConfig: { skipFileLoading: true },
});
```

:::

## Startup Phases

The `bootstrap()` function orchestrates 8 critical startup phases:

1. **Environment Detection**: Detects CI/CD vs local environment
2. **Smart .env Loading**: Opt-in .env file loading with auto-creation
3. **Port Determination**: Resolves port from priority chain
4. **Package.json Extraction**: Reads app name and version (cached)
5. **DI Container Initialization**: Creates application instance
6. **Environment Injection**: Injects environment into app instance
7. **API Version Detection**: Detects API versions from decorators
8. **Server Startup**: Starts server with graceful shutdown handling

## CI/CD Support

Bootstrap automatically detects CI/CD environments and adjusts behavior accordingly, providing zero-configuration support for containerized deployments.

### Automatic Detection

Bootstrap detects CI/CD environments by checking for platform-specific environment variables:

```typescript
// Generic CI flag (most platforms set this)
process.env.CI;

// Platform-specific detection
process.env.GITHUB_ACTIONS; // GitHub Actions
process.env.GITLAB_CI; // GitLab CI
process.env.JENKINS_URL; // Jenkins
process.env.CIRCLECI; // CircleCI
process.env.TRAVIS; // Travis CI
process.env.BUILDKITE; // Buildkite
process.env.AZURE_HTTP_USER_AGENT; // Azure DevOps
process.env.BAMBOO_BUILDKEY; // Bamboo
process.env.TEAMCITY_VERSION; // TeamCity
```

**Supported Platforms:**

-   GitHub Actions
-   GitLab CI
-   Jenkins
-   CircleCI
-   Travis CI
-   Buildkite
-   Azure DevOps
-   Bamboo
-   TeamCity

### CI/CD Behavior

When a CI/CD environment is detected:

1. **Skips .env file loading**: Uses `process.env` variables injected by the platform
2. **Validates required variables**: Checks that all required variables are present in `process.env`
3. **Never creates template files**: Prevents file creation in CI environments
4. **Provides platform-specific hints**: Error messages include setup instructions for the detected platform
5. **Supports .env.vault**: Automatically loads encrypted `.env.vault` files if `DOTENV_KEY` is present

### Platform-Specific Error Messages

When validation fails in CI/CD, bootstrap provides platform-specific setup instructions:

**GitHub Actions:**

```
🔧 GitHub Actions Setup:
   - Go to: Settings → Secrets and variables → Actions
   - Add repository secrets for: DATABASE_URL, JWT_SECRET
   - Use: ${{ secrets.VARIABLE_NAME }} in workflow files
```

**GitLab CI:**

```
🔧 GitLab CI Setup:
   - Go to: Settings → CI/CD → Variables
   - Add CI/CD variables for: DATABASE_URL, JWT_SECRET
   - Use: $VARIABLE_NAME in .gitlab-ci.yml
```

**Jenkins:**

```
🔧 Jenkins Setup:
   - Configure: Manage Jenkins → Credentials
   - Add credentials for: DATABASE_URL, JWT_SECRET
   - Use: env.VARIABLE_NAME in pipeline
```

### Performance

-   **Detection**: Cached after first call (O(1) subsequent calls)
-   **Platform identification**: Uses optimized Map lookup for O(1) performance
-   **No file I/O**: Skips file system operations in CI environments

## Performance

-   Startup time: ~8-25ms (optimized)
-   Package.json read: Cached after first call
-   CI detection: Cached after first call
-   Logger instances: Lazy initialization

## Container Development

ExpressoTS provides a seamless containerized development experience with hot reload, debugging support, and automatic configuration.

### Quick Start

The simplest way to start development in a Docker container:

```bash
expressots dev --container
```

This single command automatically:

1. **Generates Docker files** if missing (`Dockerfile.development`, `docker-compose.development.yml`)
2. **Sets up local dependencies** if your project has local file dependencies
3. **Starts the development container** with hot reload enabled

### Container Development Options

```bash
# Start container development (auto-generates files if needed)
expressots dev --container
expressots dev -c

# Rebuild container before starting
expressots dev --container --build
expressots dev -c -b

# Run in background (detached mode)
expressots dev --container --detach
expressots dev -c -d

# Combine options
expressots dev -c -b -d
```

### What's Included

When you run `expressots dev --container`, the development environment provides:

| Feature          | Description                                   |
| ---------------- | --------------------------------------------- |
| 🔄 Hot Reload    | File changes automatically restart the app    |
| 🔍 Debug Port    | Node.js debugger available on port 9229       |
| 📦 Volume Mounts | Source code synced between host and container |
| 🗄️ Services      | Auto-configured Redis, PostgreSQL if detected |
| 📊 Health Checks | Automatic container health monitoring         |

### Manual Container Configuration

For more control, you can use the `containerize` command directly:

```bash
# Generate Docker configuration interactively
expressots containerize docker

# Generate development environment
expressots containerize docker --env development

# Generate with specific preset
expressots containerize docker --preset minimal
expressots containerize docker --preset standard
expressots containerize docker --preset full

# Include CI/CD pipeline
expressots containerize docker --ci
```

### Container Presets

| Preset     | Description                                 |
| ---------- | ------------------------------------------- |
| `minimal`  | Basic Dockerfile only                       |
| `standard` | Dockerfile + Docker Compose + .dockerignore |
| `full`     | All files + Kubernetes manifests + CI/CD    |

### Generated Files

Depending on the preset, the containerize command generates:

```
project/
├── Dockerfile                      # Production multi-stage build
├── Dockerfile.development          # Development with hot reload
├── docker-compose.yml              # Production compose
├── docker-compose.development.yml  # Development compose with volumes
├── .dockerignore                   # Optimized ignore patterns
├── docker-setup.js                 # Local dependency setup script
└── k8s/                            # Kubernetes manifests (full preset)
    ├── deployment.yaml
    ├── service.yaml
    ├── configmap.yaml
    └── kustomization.yaml
```

### Local Dependencies

If your project uses local file dependencies (e.g., monorepo with `file:../shared`), the CLI automatically:

1. Creates `.docker-deps/` directory with packed dependencies
2. Generates `package.docker.json` with correct paths
3. Updates `Dockerfile.development` to use these files

```bash
# Manually run setup if needed
npm run docker:setup
```

### Stopping Containers

```bash
# Stop development containers
docker-compose -f docker-compose.development.yml down

# Stop and remove volumes
docker-compose -f docker-compose.development.yml down -v
```

### Container Profiling

Analyze your Docker setup for optimization opportunities:

```bash
# Profile Dockerfile
expressots profile container

# Profile built image
expressots profile image my-app:latest

# Generate optimization recommendations
expressots profile optimize
```

### CI/CD Integration

Generate CI/CD pipelines for your containerized application:

```bash
# Generate GitHub Actions workflow
expressots cicd generate github

# Generate for multiple platforms
expressots cicd generate all

# Include security scanning
expressots cicd generate github --security

# Include E2E tests
expressots cicd generate github --e2e
```

Supported CI/CD platforms:

-   GitHub Actions
-   GitLab CI
-   CircleCI
-   Jenkins
-   Bitbucket Pipelines
-   Azure DevOps

## Template System

ExpressoTS CLI v4 introduces a dynamic template system that enables community contributions and keeps generated code up-to-date without requiring CLI updates.

### How It Works

1. **Remote Templates**: Templates are fetched from GitHub (`expressots/templates`)
2. **Local Cache**: Cached for 24 hours in `~/.expressots/cache/templates/`
3. **Automatic Updates**: Run `expressots templates update` to refresh
4. **Embedded Fallback**: CLI always works offline with built-in templates

### Managing Templates

```bash
# List available templates
expressots templates list

# Update to latest versions
expressots templates update

# Check template system status
expressots templates status

# Use custom template repository
expressots templates repo set myorg/templates
```
